# SN13 Data Value and Incentives

## Purpose

This document defines how Jarvis reasons about data value on SN13.

Operators need explicit instructions on what to scrape. Jarvis generates those instructions from the same incentives that affect miner rewards:

- freshness
- Dynamic Desirability
- source weighting
- uniqueness and duplication
- credibility
- export/verification readiness

## Reward-Relevant Signals

### Freshness

Default policy:

- data is treated as normally scorable within a 30-day freshness window
- data older than the default freshness window is accepted as non-scorable unless a desirable job range makes it valuable

Implementation:

- `subnets/sn13/policy.py`
- `SN13Policy.classify_entity`

### Dynamic Desirability

Dynamic Desirability describes what the subnet currently wants miners to collect.

Each job contains:

- platform/source
- label
- optional keyword
- job weight
- optional start date
- optional end date

If a desirable job has a date range, data inside that range is valuable for the job path. Data outside that range is not valuable for that job path.

Implementation:

- `subnets/sn13/gravity.py`
- `subnets/sn13/desirability.py`
- `DesirabilitySnapshot`
- `DesirabilityJob`

Operational rule:

- Jarvis refreshes the real public Gravity aggregate before planning work.
- Built-in desirability samples are allowed only for tests and local development.
- A missing Gravity cache blocks operational DD planning until `jarvis-miner sn13 dd refresh` succeeds or a real `--dd-file` is supplied.

Upstream references:

- `https://github.com/macrocosm-os/data-universe/blob/main/docs/dynamic_desirability.md`
- `https://github.com/macrocosm-os/data-universe/blob/main/dynamic_desirability/data.py`

### Source Weight

SN13 scoring gives different value to different data sources.

Jarvis stores source weights in policy instead of hardcoding them throughout the system. This keeps source weights configurable when upstream changes.

Implementation:

- `SN13Policy.source_weights`

### Duplication

Duplicate data reduces value and can damage miner quality.

Jarvis uses normalized URI as the primary duplicate key. Duplicate operator submissions are recorded as `duplicate_observations` and rejected before they overwrite canonical miner truth.

Implementation:

- `SQLiteStorage.uri_exists`
- `SQLiteStorage.record_duplicate`
- `SubmissionQualityChecker.assess`

### Credibility

Validator checks affect miner credibility. Credibility strongly affects score, so Jarvis must optimize for correctness before volume.

Implementation:

- `CredibilityPolicy`
- `SubmissionQualityChecker`
- `operator_quality_stats`

Jarvis does not let operators bypass quality checks because validator penalties land on Jarvis.

## How Jarvis Turns Incentives Into Operator Work

```text
Real Gravity cache
        +
Policy constants
        +
Current SQLite coverage
        |
        v
SN13Planner
        |
        v
Ranked OperatorDemand
        |
        v
OperatorTask
```

The planner prioritizes:

- high desirability weight
- underfilled buckets
- recent buckets
- configured source weight

The planner suppresses:

- fully covered buckets
- buckets outside desirable date windows
- sources not enabled in the current planner configuration

Current source publication gate:

- X and Reddit are enabled for operator work.
- Other Gravity sources may be visible in `dd show`, but they are not published into the workstream until Jarvis has confirmed source payload validation, export schema, and readiness gates for that source.

## How Data Is Classified

Every submission is classified as one of:

- `accepted_scorable`
- `accepted_non_scorable`
- `rejected`

### Accepted Scorable

The submission is valid and currently valuable under policy or desirability rules.

### Accepted Non-Scorable

The submission is valid, but not currently valuable. Example: data outside the default freshness window with no desirable override.

Jarvis stores this separately from rejection because valid historical data can still be useful for export, replay, or future analysis.

### Rejected

The submission is not allowed into miner truth.

Reasons include:

- duplicate entity
- missing source-specific fields
- source payload mismatch
- empty content

## Operator Implication

Operators do not scrape random data.

Operators execute tasks generated by Jarvis because those tasks encode:

- current desirability
- source priority
- bucket coverage gaps
- freshness requirements
- quantity targets
- expiry
